## 2. Цели дизайна и ограничения

**Единственный прямой проход.** Полный парсер (`parse_text!`) сканирует
источник ровно один раз, слева направо. Нет бэктрекинга и нет выделения
памяти в куче для состояния парсера — состояние активных блоков и
унифицированный ограниченный стек вложенности инлайн-движка (один стек,
разделяемый правилами symmetric, asymmetric и `key_value`) — это
фиксированного размера массивы на стеке, размер которых задаётся настройкой
`max_nest` (см. §9, §11, §17). Инлайн-сканирование может идти по
многострочному fallthrough-прогону, а не по одной строке (§7, §9), но
по-прежнему стримит строго вперёд — прогон накапливается, не пересканируется.

**Плоский SoA-вывод.** Контент-структура хранит один `Vec` на каждый вид
элементов. Все спаны — `u32` байтовые смещения в исходный срез, который
заимствуется на время жизни контента. Такой лейаут даёт O(1) доступ на вид
элементов и дружелюбен к кешу при последующей обработке.

**Декларативная грамматика, ноль рукописного сканирования.** Автор грамматики
пишет один вызов `define_parser!`. Каждый метод парсера — `parse`, `find_*`,
`_clean`, `_raw` — генерируется во время компиляции. Нет диспатча в рантайме
и нет vtable.

**Stable-first.** По умолчанию крейт компилируется на стабильном Rust.
SIMD-ускорение подключается опционально через feature-флаги Cargo, требующие
nightly.

**Без unsafe.** Крейт не содержит кода `unsafe`. Арифметика спанов использует
`saturating_sub` и `.min(len)` на границах, где иначе возможно переполнение
или выход за пределы.

## 3. Разделение на два крейта

Движок разделён на два крейта по структурной причине: крейты процедурных
макросов должны компилироваться как библиотека особого рода (dylib,
загружаемая компилятором), и они не могут экспортировать обычные элементы
вместе с `#[proc_macro]`-точками входа.

```
meon-macros   — крейт процедурного макроса
  define_parser!  (единственная публичная точка входа)
  ↓  испускает токены, вызывающие:
meon          — крейт рантайма
  define_content!         (декларативный макрос, #[macro_export])
  parse_text!             (декларативный макрос, #[macro_export])
  parse_inline!           (декларативный макрос, #[macro_export])
  parse_line!             (декларативный макрос, #[macro_export])
  parse_block!            (декларативный макрос, #[macro_export])
  define_standalone_fns!  (декларативный макрос, #[macro_export])
  *Iter структуры         (реализации standalone-итераторов)
```

Авторы грамматик зависят от `meon` (который ре-экспортирует `define_parser!`
из `meon-macros`) и только от него. Разделение невидимо снаружи.

## 4. Конвейер компиляции грамматики

Когда компилятор разворачивает `define_parser!(Name { ... })`, следующие стадии
выполняются последовательно, полностью во время компиляции:

```
Исходные токены (грамматический DSL)
        │
        ▼
  [cursor.rs] рукописный курсор TokenStream
        │  обходит поток токенов посекционно (включая опциональное
        │  значение контекста max_nest, наряду с sep/eol/tab/escape)
        ▼
  [collect.rs] фронтенд грамматики
        │  заполняет CF (collected fields) и Vec<StandaloneRule>
        ▼
  [strip.rs] хирургия токенов
        │  удаляет аннотации => field [N] из секций грамматики,
        │  чтобы очищенные токены можно было передать в рантайм-макросы
        ▼
  [codegen.rs] бэкенд: эмиссия контент-структуры
        │  испускает: define_content!(Name { ... })
        ▼
  [methods.rs] бэкенд: эмиссия акцессоров
        │  испускает: impl<'a> NameContent<'a> { str, bytes, *_clean, *_raw }
        ▼
  [codegen.rs] бэкенд: эмиссия standalone DSL
        │  испускает: define_standalone_fns! { ... }
        ▼
  Итоговый токен-стрим передаётся обратно rustc
```

Сам процедурный макрос (`expand` в `lib.rs`) тонкий: управляет курсором,
вызывает три функции сбора, вызывает strip, и собирает вывод из трёх
строителей бэкенда.

## 5. Промежуточное представление

Собранные поля (CF) — простая структура, накапливаемая фронтендом:

```rust
struct CF {
    inline:        Vec<(Ident, TokenStream, Literal)>,  // поле, Type, div
    inline_simple: Vec<(Ident, Literal)>,               // поле, div
    line:          Vec<(Ident, TokenStream, Literal)>,
    block:         Vec<(Ident, TokenStream, Literal)>,
    block_simple:  Vec<(Ident, Literal)>,
    standalone:    Vec<StandaloneRule>,
}
```

`StandaloneRule` — перечисление с одним вариантом на каждую грамматическую
конструкцию, поддерживающую standalone-сканирование:

```
SymmetricExact  { field, byte, count }
AsymmetricExact { field, open, close, count }
Chained         { field, open1, close1, open2, close2, prefix, ty, pf, ff, sf }
KeyValue        { field, eq, end, allow_sep, ty, kf, vf }
LineMarker      { field, byte, max, ty, var, body }
LineUniform     { field, bytes, min, ty, var, body }
Fence           { field, byte, min }
Cont            { field, byte }
BlockMarker     { field, bytes, ty, var, body }
BlockNumbered   { field, end_bytes, ty, num_var, kind_var, body }
```

Каждый вариант несёт ровно те литералы и идентификаторы, которые нужны для
эмиссии метода `find_*` (через `define_standalone_fns!`) и акцессоров
`_clean` / `_raw` (через `methods.rs`). Ничего лишнего не хранится.

Заметим, что `max_nest` сам по себе *не* является частью `CF` — он разбирается
напрямую в `expand()` (`lib.rs`) как отдельный `Literal`, наряду с
`sep`/`eol`/`tab`/`escape`, и передаётся прямо в точку вызова `parse_text!`,
а не пропускается через структуру собранных полей.

## 6. Кодогенерация

### 6.1 `define_content!`

`build_define_content` испускает единственный вызов рантайм-макроса:

```
mc::define_content!(Name {
    inline        { field: Type [div], ... }
    inline_simple { field [div], ... }
    line          { field: Type [div], ... }
    block         { field: Type [div], ... }
    block_simple  { field [div], ... }
});
```

`mc` — разрешённый путь к рантайм-крейту (`crate` при разворачивании внутри
самого `meon`, импортированное имя в других случаях). Механизм гигиены описан
в §16.

`define_content!` сам (декларативный макрос в `engine/content.rs`) разворачивает
это в два типа:

- `NameState` — изменяемый аккумулятор, все поля `pub(crate)`.
- `NameContent<'a>` — публичная выходная структура, заимствующая источник.

Также испускаются методы `push_<field>` и `push_merge_<field>` на `NameState`
для каждого `inline_simple`-поля. Вариант merge объединяет смежные спаны
(зазор ≤ 1 байт) в одну запись.

### 6.2 Структура парсера и метод `parse`

Разворачивание `define_parser!` испускает:

```rust
pub struct NameParser;

impl NameParser {
    pub fn parse(source: &[u8]) -> NameContent<'_> {
        mc::parse_text!(
            source;
            sep = ..., eol = ..., tab = ..., escape = ..., max_nest = ...;
            inline  { /* очищенная грамматика */ }
            lines   { /* очищенная грамматика */ }
            blocks  { /* очищенная грамматика */ }
        )
    }

    mc::define_standalone_fns! { sep=..., eol=..., tab=..., escape=...; ... }
}
```

`max_nest` всегда явно передаётся в `parse_text!`, даже если грамматика его
не указала — `expand()` (`lib.rs`) дефолтит литерал в `1` перед эмиссией этого
вызова, не полагаясь на собственную ветвь дефолта `parse_text!`.

Секции грамматики, переданные в `parse_text!`, были очищены от аннотаций
`=> field [N]` с помощью `strip.rs` — эти аннотации значимы только для
фронтенда процедурного макроса и были бы синтаксическими ошибками внутри
рантайм-макросов.

### 6.3 Методы-акцессоры

`build_content_methods` испускает по одной паре `_clean` / `_raw` на каждый
`StandaloneRule` и по одному акцессору `_clean` для каждого простого поля
без standalone-правила.

`_clean` возвращает срез внутреннего содержимого (между разделителями).
`_raw` возвращает полный срез включая байты разделителей, используя
`saturating_sub` на нижней границе и `.min(source.len())` на верхней, чтобы
предотвратить выход за пределы.

## 7. Архитектура рантайма

`parse_text!` — многостадийный декларативный макрос, работающий в фазах
накопления во время компиляции перед эмиссией фактического O(n) цикла
парсинга.

### Стадии накопления (время компиляции)

```
parse_text!(src; sep=..., eol=..., tab=..., escape=..., max_nest=...; <секции>)
    │
    ├─ @cs  — разбить сырые секции на типизированные корзины [inline], [lines], [blocks]
    │
    ├─ @ci  — обойти корзину inline:
    │         извлечь флаг merge_simple
    │         извлечь имя поля fallback
    │         извлечь правило hard_break
    │         собрать наборы байт on_trigger → список finders
    │         собрать тела on_trigger → ilt (inline token list)
    │
    ├─ @cb  — обойти корзину blocks:
    │         извлечь правила block_simple → sr (simple rules)
    │         извлечь правила block        → br (block rules)
    │         извлечь поле параграфа fallback
    │
    └─ @body — испустить фактический цикл парсинга
               со всеми разрешёнными корзинами, встроенными как tt-фрагменты
```

Каждая стадия — отдельная ветвь макроса, идентифицируемая внутренним токеном
с префиксом `@`. Финальная ветвь `@body` получает всё как плоские `tt`-фрагменты
и испускает единственное блочное выражение Rust, являющееся телом `parse`.

### Основной цикл (рантайм)

```
pos = 0
while pos < len:

    if в начале строки:
        if пустая строка → сбросить параграф, закрыть активные фреймы блока, продвинуться
        найти current_line_end

        loop:
            попробовать parse_block! ветвь active
                → Some((false, cs))  — активный стек потребил/продолжил эту строку
            попробовать parse_block! ветвь open
                → Some((true,  cs))  — открылся новый блок (возможно несколько, вложенных)
            попробовать parse_line!
                → Some(cs)           — совпало строчное правило
            если глубина активного стека изменилась без совпадения (внешнее
                продолжение только что закрылось в середине строки) →
                повторить с того же начала строки против теперь более
                неглубокого стека
            break если нет прогресса

        если любое из вышеперечисленного совпало → перейти к следующей строке

    if самый внутренний активный фрейм — ограждение → пропустить к следующей строке (без инлайн-скана)

    if строка не совпала ни с line/block-правилом и нет активного блока:
        отложить — записать/продлить текущий fallthrough-прогон (para_start) и
        перейти к следующей строке, ПОКА не сканируя её инлайном
    else (хвостовой контент после совпавшего line/block-маркера):
        просканировать этот контент одним parse_inline!, ограниченным строкой

    отложенный fallthrough-прогон сбрасывается ОДНИМ многострочным вызовом
    parse_inline! — и записывается его спан параграфа — когда закрывается: на
    пустой строке, на строке где совпадает line/block-правило, или в конце
    ввода (см. §9, «Многострочные прогоны»)
```

Инвариант цикла: `pos` всегда продвигается. Fallthrough-прогон сканируется
один раз на всю свою многострочную протяжённость, а не построчно, поэтому
унифицированный инлайн-стек сохраняется через байты `\n` внутри него. Проверка
hard-break и сброс текста происходят на границах прогона, не внутри
посимвольного цикла `parse_inline!`.

## 8. Контент-структура и аккумулятор состояния

Контент-структура имеет пять категорий полей, каждая с отдельным лейаутом
хранения, непосредственно отражающим семантику парсинга:

| Секция         | Тип поля                | Заполняется через   |
|----------------|--------------------------|---------------------|
| `inline`       | `Vec<T>`                | `parse_inline!`     |
| `inline_simple`| `Vec<Span>`             | `parse_inline!`     |
| `line`         | `Vec<(T, Span)>`        | `parse_line!`       |
| `block`        | `Vec<(T, Span)>`        | `parse_block!`      |
| `block_simple` | `Vec<Span>`             | `parse_block!`      |

`Vec<T>` — пользовательский тип с несколькими `Span`-полями (ссылки, пары ключ-значение).
`Vec<Span>` — один байтовый диапазон, нет дополнительных метаданных.
`Vec<(T, Span)>` — метаданные на элемент в паре с контентным спаном.

Аккумулятор `NameState` предварительно выделяется с ёмкостью
`source.len() / div` для каждого поля. По окончании парса `into_content(source)`
перемещает все векторы в публичную структуру и прикрепляет ссылку на источник.
Данные спанов не копируются.

## 9. Инлайн-парсинг

`parse_inline!` выполняет однопроходное сканирование одного **прогона**
источника — либо одной строки (хвостовой контент после совпадения
`line`/`block`), либо целого многострочного fallthrough-спана, переданного из
`parse_text!` одним вызовом (см. §7 и «Многострочные прогоны» ниже). Напрямую
вызывать не предполагается.

### Фазы накопления (время компиляции)

Макрос собирает секцию `inline { ... }` в типизированные корзины:

```
finders  — наборы байт on_trigger (байт `end` каждого key_value авто-добавлен)
sy_rules — симметричные правила
as_rules — асимметричные правила
ch_rules — chained-правила
kv_rules — правила key_value
hb       — правило hard_break (не более одного)
```

Затем переходит к ветви `@body` со всеми корзинами, сплющенными в tt-фрагменты.

### Унифицированный стек вложенности

До появления вложенности symmetric использовал один pending-слот, asymmetric —
один форвард-поиск. Теперь движок держит **каждую stack-eligible конструкцию —
symmetric, asymmetric И key_value — на одном общем стеке**, ограниченном
грамматическим `max_nest` (передаётся из `parse_text!`; по умолчанию `1`):

```
frames:        [(u8, u8, u32); max_nest]   // (byte, count, vidx)
fdepth:        usize                        // единый бюджет для ВСЕХ видов
kv_pending:    [(u32, u32, u32); max_nest]  // (key_start, key_end, value_start)
asym_overflow: u32                          // одноразовый счётчик для balanced-открытий сверх предела
```

Фрейм хранит только `(byte, count, vidx)` — **тега вида нет**. Вид фрейма, его
закрывающий байт, непрозрачность и маршрутизация в поле восстанавливаются в
момент использования сопоставлением хранимого `byte` с компайл-тайм литералом
каждого правила — тот же идиом, что open-сторона уже использует для
маршрутизации поля. Это опирается на общесистемное допущение, что байт имеет
один смысл среди stack-eligible правил блока `on_trigger`.

- **asymmetric** — `byte` = открывающий байт правила; `count` = всегда 1
  (каждый байт открывающего прогона — своё событие); `vidx` = индекс
  placeholder-спана, запушенного при открытии и back-patch'нутого при закрытии.
  Непрозрачность выводится из хранимого открывающего байта, не хранится на фрейм.
- **symmetric** (`parse_inside = true, balanced = true`) — `byte` =
  разделитель; `count` = длина прогона (выбирает поле 1/2/3 →
  italics/bolds/bold_italics и точно сверяется при закрытии); `vidx` = индекс
  placeholder'а.
- **key_value** — `byte` = байт `end` правила; `count` = 0 (не используется);
  `vidx` = 0 (не используется). Pending-данные лежат в `kv_pending` на
  собственной позиции фрейма в стеке. В отличие от двух других, kv-пара **не**
  пушится при открытии; она дописывается целиком только когда её значение
  закрывается — это держит `Vec<T>` в порядке закрытия значений (внешняя пара
  ложится после всех вложенных в её значение), противоположно open-order
  конвенции asymmetric/symmetric для их placeholder-at-open векторов.

`fdepth` — единый бюджет для всех трёх видов; любая проверка «открыто ли
что-нибудь» — это `fdepth == 0`, независимо от того, какие виды на стеке.

### Конструкции вне стека (исходные пред-nesting пути, нетронуты)

Они никогда не трогают унифицированный стек и выполняют свой исходный код:

- **symmetric `parse_inside = true, balanced = false`** — единственный
  `pending`-слот.
- **symmetric `parse_inside = false`** (жадный, кодовые спаны) — форвард-поиск;
  получил только escape-осведомлённость.
- **asymmetric `balanced = false, parse_inside = false`** (автолинки) —
  `memchr`/поиск-глубины; его закрывающий байт в `on_trigger` НЕ требуется.
- **chained** — двухфазная прозрачная машина состояний (или исходный
  непрозрачный двухфазный форвард-поиск). Фазы строго последовательны — фаза 2
  начинается только когда фаза 1 полностью закрылась — поэтому достаточно
  одного слота на фазу: без стека, без расхода `max_nest`.

### Закрытие — единый проход с дренажом key_value

Закрывающий байт выполняет **один** проход, не отдельный блок на правило (что
дважды снимало бы общий закрывающий байт). На каждый закрывающий символ:
сначала дренируется любой `key_value`-фрейм на верхушке (его контейнер
закрывается, значит значение кончается здесь, фиксируется до pop'а контейнера);
затем, если новая верхушка — asymmetric-фрейм, чей грамматически-известный
закрывающий байт совпадает с символом, он снимается и его placeholder
back-patch'ится. Диспатч — по **собственному записанному открывающему байту**
фрейма, так что два правила с общим закрывающим байтом (`(`/`)` и `[`/`)`)
закроют правильный фрейм. Поскольку стек — строгий LIFO, а kv-фрейм всегда выше
контейнера, в котором лежит, прогон `}}` финализирует внутреннюю пару, затем её
объект, затем внешнюю пару — корректная вложенность выпадает из посимвольного
цикла.

Грамматическое следствие: как только правило на стеке, его закрывающий байт (и
`end` у `key_value`) обязан быть в том же `on_trigger`, поскольку закрытия
ищутся тем же `find_any`-сканом, что и открытия. `end` у `key_value`
авто-добавляется; закрывающий байт asymmetric пока нужно указывать вручную
(см. §17).

### Дренаж в конце прогона

Фрейм, всё ещё открытый в конце прогона, дренируется сверху вниз, вид
восстанавливается по хранимому байту:

- **key_value** — пушится целиком, значение финализируется до конца прогона
  (так что `key = value` без терминатора всё равно эмитится); текстовый курсор
  продвигается за него, чтобы финальный сброс текста не переэмитил значение.
- **asymmetric** — отбрасывается через `Vec::remove(vidx)` (закрытая
  внутренняя само-вкладывающаяся запись может стоять на *более высоком* индексе,
  чем ещё открытая внешняя).
- **symmetric** — отбрасывается через `truncate(vidx)` (идентичная
  `(byte, count)` никогда не само-вкладывается, поэтому на поле максимум один
  живой placeholder, всегда последний).

### Переполнение предела

Сверх `max_nest` дополнительное открытие того же типа **asymmetric** с
`balanced = true` инкрементирует `asym_overflow` вместо push'а; следующее
закрытие того же типа потребляет одну единицу overflow вместо pop'а, так что
закрытие реально отслеживаемого фрейма не принимается слишком рано. Открытие
**symmetric** сверх предела и `eq` у **key_value** сверх предела просто
поглощаются (фрейма нет, пара/прогон не отслеживаются).

### Многострочные прогоны

`parse_text!` может передать `parse_inline!` спан на несколько строк одним
вызовом (§7). Внутри такого прогона `eol` — обычное содержимое: унифицированный
стек переживает его, поэтому открытый контейнер или незавершённое значение
`key_value` переживают перенос строки и дренируются только в настоящем конце
прогона. `eol` входит в унифицированный триггер-сет `find_any` **только** когда
многострочный вызов передаёт `multiline = true`; вызов, ограниченный строкой,
передаёт `false` (его спан по построению без `\n`, так что искать там `eol`
было бы мёртвым сравнением на каждый chunk). Когда объявлен `hard_break`,
внутренний `\n` в многострочном прогоне проверяется на hard-break так же, как и
конец прогона.

### Диспатч `find_any`

Поиск триггера — один вызов `swar::find_any` с константным по размеру массивом,
мономорфизируемый во время компиляции:

```
N=1 → memchr::memchr
N=2 → memchr::memchr2
N=3 → memchr::memchr3
N≥4 → SWAR / SIMD-цикл (см. §13)
```

Несколько блоков `on_trigger` вносят свои байты; внутренний цикл берёт
минимальное смещение по результатам finders для ближайшего триггера. `eol`
включается как одна доп-цель только на многострочных вызовах (см. выше).

## 10. Строчный парсинг

`parse_line!` вызывается в начале каждой новой строки, до блочной и
инлайн-обработки. Пробует каждое правило в порядке объявления и возвращает
`Some(cs)` при первом совпадении, где `cs` — байтовое смещение первого
контентного байта (для `line`) или конца строки (для `line_simple`).

**`line(byte, max = N)`:** подсчитывает последовательные байты `byte` от
`pos`. Если счёт в диапазоне [1, max] и за ним следует `sep` или конец
строки — совпадение записывается. Спан покрывает всё после маркера и его sep.

**`line_simple(b1 | b2 | ..., min = N)`:** читает первый байт. Если он
совпадает с паттерном — валидируется вся строка: каждый байт должен быть
либо тем же разделителем, либо `sep`. Если валидна и счёт разделителя ≥ min —
совпадение записывается.

Оба правила используют простой фаллтроу: если первое правило не совпало,
пробуется следующее. Если ни одно не совпало — возвращается `None` и
`parse_text!` переходит к блочной/инлайн-обработке.

## 11. Блочный парсинг

`parse_block!` разрешает полную блочную структуру строки за один вызов,
через две фазы:

### Фаза снятия (peel)

Каждый текущий открытый фрейм сопоставляется со строкой снаружи внутрь:
маркеры продолжения потребляются, ещё открытое ограждение либо закрывается,
либо поглощает строку, а фрейм чей маркер исчез закрывается — вместе со
всем, что вложено внутри него.

### Открывающая фаза (open)

На позиции, оставшейся после снятия, открываются новые фреймы — столько,
сколько вкладывается в одной строке (например, ограждение открывающееся
внутри уже открытой цитаты) — ограниченные `max_nest`.

### Стек активных блоков (`max_nest`)

Состояние активных блоков — ограниченный стек `[(u8, u8, u8, u32);
max_nest]` плюс счётчик глубины, разделяющий грамматический предел
`max_nest` с инлайн-движком (§9). `max_nest = 1` сокращает его до
единственного слота и точно воспроизводит исходное, single-active-block
поведение: максимум один блок открыт за раз, никакой блок не открывается
внутри другого.

| Дискриминант (поле 0) | Значение            | Поле 1   | Поле 2  | Поле 3   |
|------------------------|----------------------|----------|---------|----------|
| `0`                    | Открытое ограждение  | `byte`   | `count` | `start`  |
| `1`                    | Продолжение (`>`)    | `byte`   | `0`     | `start`  |

Два структурных инварианта делают стек tractable:

- **Ограждение всегда верхний фрейм.** Содержимое ограждения непрозрачно —
  никакой блок не может открыться внутри него — так что когда ограждение
  самый внутренний открытый блок, оно потребляет всю строку и открывающая
  фаза никогда не запускается; ничего никогда не push'ится поверх
  ограждения.
- **Продолжения могут само-вкладываться.** В отличие от инлайн-симметричного
  разделителя (где открытие и закрытие неотличимы, так что идентичный ключ
  не может вложиться), `cont` открывается позиционно — в начале строки,
  после того как внешние маркеры были сняты — и закрывается по
  *отсутствию* своего маркера, так что `> >` — это два по-настоящему
  вложенных фрейма цитаты.

`block`-элементы (списки, нумерованные списки) — листья на каждую строку:
они ничего не push'ят в стек и поэтому не потребляют глубину. Они всё ещё
могут открыться *внутри* `cont`, но только когда `max_nest > 1` — при
`max_nest = 1` открывающая фаза никогда не запускается внутри уже открытого
блока, точно как раньше.

Если маркер внешнего продолжения исчез в середине строки пока внутренние
фреймы были ещё открыты, эти внутренние фреймы закрываются вместе с ним
(начиная с самого внутреннего), глубина падает, и основной цикл
`parse_text!` повторяет с того же начала строки, так что остаток строки
переобрабатывается заново против теперь более неглубокого стека — см.
псевдокод основного цикла в §7.

### Возвращаемое значение

`Some((true, cs))` — открылся хотя бы один новый блок; `cs` — первый
контентный байт (или начало следующей строки, для ограждения чья строка
с инфо потребляется целиком). `Some((false, cs))` — активный стек
потребил/продолжил эту строку без открытия нового. `None` — ничего не
совпало и ничего не активно, **или** внешнее продолжение только что
закрылось; в последнем случае вызывающий повторяет с того же начала строки
чтобы переобработать остаток.

## 12. Standalone-итераторы

Каждое правило, поддерживающее standalone-сканирование, генерирует метод
`find_*` через `define_standalone_fns!`. Каждый метод конструирует одну из
структур итераторов из `engine/text_parser/standalone/`.

Все standalone-итераторы разделяют один контракт:

- Сканируют сырой источник независимо.
- Не несут перекрёстного состояния элементов (нет стека активных блоков,
  нет отслеживания параграфов, нет диспатча инлайн-триггеров, и нет
  ограниченного стека вложенности какого-либо вида — standalone-правила
  `symmetric`/`asymmetric` совпадают только с *точным* объявленным счётом,
  полностью игнорируя `balanced`/`parse_inside`).
- Могут совпадать с байтами, которые `parse_text!` подавил бы (например,
  разделитель внутри ограждённого блока).
- Их вывод может отличаться от полного парса — намеренно.

### Структура итератора

Каждый итератор хранит параметры, переданные в его конструктор `new`, плюс
курсор `pos` и, для итераторов ограниченных строкой, курсор `line_end`.

Метод `next` циклически: продвигает `line_end` при исчерпании, находит
следующего кандидата через `memchr`, валидирует кандидата, возвращает
`Some(span)` при успехе или продолжает цикл при неудаче.

Итераторы используют три общие утилиты из `standalone/common.rs`:

- `find_line_end(src, from, eol)` — найти конец текущей строки.
- `advance_line(src, line_end, eol)` — перейти к следующей строке, вернуть `None` при EOF.
- `count_escape(src, pos, escape)` — подсчитать последовательные байты escape перед `pos`, для обнаружения экранированных разделителей.

### Типы итераторов

| Тип                  | Соответствующее правило | Тип элемента    |
|-----------------------|---------------------------|-----------------|
| `SymmetricExactIter` | `symmetric N =>`        | `Span`          |
| `AsymmetricExactIter`| `asymmetric N =>`       | `Span`          |
| `ChainedIter`        | `chained`               | `T`             |
| `KvIter`             | `key_value`             | `T`             |
| `LineMarkerIter`     | `line`                  | `(T, Span)`     |
| `LineUniformIter`    | `line_simple`           | `(T, Span)`     |
| `FenceIter`          | `fence`                 | `Span`          |
| `ContIter`           | `cont`                  | `Span`          |
| `BlockMarkerIter`    | `block (pattern)`       | `(T, Span)`     |
| `BlockNumberedIter`  | `block num(...)`          | `(T, Span)`     |

## 13. Многобайтовый поиск — SWAR и SIMD

Модуль `swar` предоставляет `find_any<const N>` — единственный примитив
поиска, используемый во всём движке.

### Диспатч

```rust
pub fn find_any<const N: usize>(targets: [u8; N], src: &[u8]) -> Option<usize> {
    match N {
        1 => memchr::memchr(targets[0], src),
        2 => memchr::memchr2(targets[0], targets[1], src),
        3 => memchr::memchr3(targets[0], targets[1], targets[2], src),
        _ => find_any_wide(targets, src),
    }
}
```

`match N` свёртывается во время мономорфизации — ветви в рантайме не
испускаются. Для N=1..3 всё обрабатывает высокооптимизированный крейт
`memchr`. Для N≥4 используется собственный `find_any_wide` движка.

### SWAR-фаллбек (stable, по умолчанию)

Для N≥4 `find_any_wide` обрабатывает 8 байт за итерацию через одно `u64`-слово:

```
ONES  = 0x0101_0101_0101_0101   (множитель трансляции)
HIGHS = 0x8080_8080_8080_8080   (маска старших битов)

broadcast(b)          = b as u64 * ONES
has_byte(chunk, cast) = (chunk ^ cast).wrapping_sub(ONES) & !(chunk ^ cast) & HIGHS
```

`has_byte` возвращает маску со старшим битом в каждом байт-лейне равном
целевому. Если какой-либо лейн совпадает — количество завершающих нулей,
делённое на 8, даёт индекс байта.

Для нескольких целей маски объединяются через OR:

```rust
let mut mask = 0u64;
for &bcast in &bcasts {
    mask |= has_byte(chunk, bcast);
}
```

Скалярный хвост обрабатывает оставшиеся < 8 байт.

### SIMD-ускорение (nightly, опционально)

С `--features avx2` или `--features avx512` крейт включает `feature(portable_simd)`
и заменяет SWAR-цикл более широким SIMD-циклом:

```
avx2:   32-байтовые Simd<u8, 32> лейны
avx512: 64-байтовые Simd<u8, 64> лейны, затем 32-байтовые, затем 8-байтовые
```

SIMD-цикл использует `simd_eq` + извлечение битовой маски. SWAR-путь
полностью компилируется прочь когда активен SIMD-feature.

## 14. Представление спанов

```rust
pub struct Span {
    pub start: u32,
    pub end: u32,
}
```

Полуоткрытый байтовый диапазон `[start, end)`. Все значения, производимые
парсером, удовлетворяют `start <= end <= source.len()`.

`u32` выбран вместо `usize` чтобы вдвое уменьшить размер спана (8 байт против
16 байт на 64-битных платформах). Это важно когда документ производит десятки
тысяч спанов — векторы занимают меньше строк кеша. Компромисс — ограничение
входа в 4 ГБ (`MAX_INPUT_LEN = u32::MAX as usize`), превышающее любой
реальный документ.

Спаны нулевой длины (`start == end`) используются как маркеры позиций, например
якоря жёстких переносов.

Fuzzing-harness явно проверяет `start <= end && end <= source.len()` для
каждого спана каждого вида элементов после каждого парса. Это основной
инвариант корректности.

## 15. Методы-акцессоры

Каждая контент-структура получает следующие методы, генерируемые `methods.rs`:

**`str(span) -> Option<&str>`** — взять срез источника по спану, валидировать
UTF-8. Возвращает `None` при невалидном UTF-8 вместо паники. Паника на
контролируемом пользователем вводе из библиотеки небезопасна.

**`bytes(span) -> &[u8]`** — взять срез источника по спану, без проверки UTF-8.

**`<field>_clean()`** — итератор над срезами внутреннего содержимого (между
разделителями).

**`<field>_raw()`** — итератор над полными срезами включая байты разделителей.
Использует `saturating_sub` на нижней границе и `.min(source.len())` на верхней
для предотвращения выхода за пределы.

Логика raw-акцессора различается по виду правила:

- `SymmetricExact`: `start - count .. end + count`
- `AsymmetricExact`: `start - count .. end + 1`
- `Chained`: `ff.start - 1 - prefix_len .. sf.end + 1`
- `BlockMarker`: `start - 2 .. end` (маркер + sep)
- `BlockNumbered`: обходит назад через sep/tab и цифровые байты

Эти акцессоры полностью отдельны от механизма ограниченной вложенности
полного парса, описанного в §9/§11 — они работают над (невложенными,
точно-счётными) данными `StandaloneRule`, согласно примечанию §12 о том, что
standalone-сканирование полностью игнорирует `balanced`/`parse_inside`.

## 16. Гигиена макросов между крейтами

Процедурный макрос испускает вызовы декларативных макросов `meon` из
произвольных крейтов-потребителей. Неквалифицированные имена макросов не
разрешились бы. Решение:

### `crate_path()`

`model.rs` разрешает имя рантайм-крейта во время разворачивания
процедурного макроса:

```rust
pub(crate) fn crate_path() -> TokenStream {
    match crate_name("meon") {
        Ok(FoundCrate::Itself)       => quote! { crate },
        Ok(FoundCrate::Name(name))   => { let i = Ident::new(&name, ...); quote! { #i } },
        Err(_)                        => quote! { crate },
    }
}
```

`proc_macro_crate::crate_name("meon")` читает `Cargo.toml` вызывающего
крейта во время компиляции и возвращает имя, под которым импортирован `meon`
(обрабатывая переименования через `package = "meon"`).

### Эмиссия

Каждый вызов макроса в генерируемом коде предваряется разрешённым путём:

```rust
let mc = crate_path();
quote! {
    #mc::define_content!(...);
    #mc::parse_text!(...);
    #mc::define_standalone_fns! { ... }
}
```

### `$crate` в декларативных макросах

Внутри `define_content!` и `parse_text!` ссылки на другие элементы крейта
`meon` используют `$crate::`:

```rust
$crate::span::Span
$crate::parse_text!(@dispatch ...)
$crate::swar::find_any(...)
```

Это стандартная гигиена декларативных макросов и работает независимо от
способа импорта `meon`.

### Внешние зависимости крейтов в разворачиваниях макросов

`paste` и `memchr` используются внутри `define_content!` и `parse_text!`
без квалификации `$crate::` — декларативные макросы не могут проксировать
пути внешних крейтов через `$crate`. Движок ре-экспортирует оба:

```rust
// meon/src/lib.rs
#[doc(hidden)] pub use paste;
#[doc(hidden)] pub use memchr;
```

И ссылается на них как `$crate::paste::paste!` и `$crate::memchr::memchr`
внутри макросов. Грамматическим крейтам достаточно зависеть только от `meon`;
`paste` и `memchr` — транзитивные зависимости.

## 17. Известные ограничения и намеренные компромиссы

### Ограниченная глубина вложенности (`max_nest`)

И стек активных блоков на блочном уровне (§11), и инлайн-уровневый
унифицированный стек вложенности — разделяемый правилами symmetric, asymmetric
И key_value (§9) — используют одну грамматическую настройку `max_nest`. Её
дефолт, `1`, точно воспроизводит исходное поведение с единственным
слотом/единственным pending: максимум один блок активен за раз, никакого
само-вложения для `balanced`-правил. Установка большего значения решает то,
что раньше было жёсткими ограничениями — цитата содержащая ограждённый код,
или цитата вложенная в другую, раньше утекала содержимым в неверный спан;
внутренний разделитель эмфазы с другим счётом раньше молча перезатирал
единственный pending-слот и полностью терял внешнюю пару.

`max_nest` — жёсткий предел, не безграничный стек — конструкции вложенные
глубже `max_nest` не отслеживаются специально:

- Для блоков, глубина за пределами `max_nest` просто прекращает открывать
  новые фреймы; пятый уровень вложенности цитаты при `max_nest = 4`
  остаётся на откуп тому, что следует дальше (как правило, обычный или
  инлайн-контент), а не представляется как пятый спан.
- Для инлайн-правил `asymmetric`/`symmetric` с `balanced = true`,
  дополнительное открытие того же типа за пределами предела инкрементирует
  одноразовый счётчик overflow и считается обычным контентом вместо
  открытия фрейма; overflow потребляется следующим закрытием того же типа,
  так что закрытие реально отслеживаемого фрейма не принимается по ошибке
  слишком рано. `eq` у `key_value` сверх предела поглощается (пара не
  отслеживается).

Компромисс тот же по духу что и исходный single-slot дизайн: маленький,
фиксированного размера массив на стеке — размером по собственному `max_nest`
грамматики, не по длине входа — полностью избегает выделения в куче и держит
обычный случай (`max_nest = 1`, дефолт) на той же стоимости что раньше, давая
грамматике опцию вложенности поглубже только там, где она реально нужна.

### `chained`-правила ограничены одним активным совпадением на грамматику

`chained`-правило с прозрачным компонентом (`parse_inside = true` на текстовой
или url-стороне) отслеживает своё незавершённое совпадение в небольшом наборе
локальных переменных, разделяемых между всеми `chained`-правилами грамматики,
а не в массиве на правило. Два `chained`-правила с перекрывающимися
незавершёнными прозрачными совпадениями делили бы то же scratch-состояние.
Каждая виденная грамматика объявляет ровно одно `chained`-правило, так что
это принимается как задокументированное ограничение, а не выстраивается
дальше.

### `key_value` ограничен одним правилом на грамматику

`key_value`-фрейм различается на унифицированном стеке только по хранимому
байту `end`, а якорь сегмента ключа и тест «верх — kv» разделяются между всеми
`key_value`-правилами. Два таких правила — особенно с общим байтом `end` — не
различимы при закрытии/дренаже, а общий якорь/тест их конфликтуют.
Поддерживается одно `key_value`-правило на грамматику. Это следствие того, что
минимальный фрейм не несёт тега вида (§9), а не фундаментальный предел:
дискриминатор на правило (например, в неиспользуемом слоте `count` kv-фрейма)
его бы снял.

### Инлайн-прогоны охватывают несколько строк, ограничены пустыми строками

`parse_inline!` получает fallthrough-прогон, который может охватывать несколько
физических строк (§9), а не одну строку. Унифицированный стек сохраняется через
внутренние байты `eol`, поэтому акцентирование, контейнер или значение
`key_value` могут пересекать перенос строки внутри одного прогона. Прогон — и
стек — ограничивается пустой строкой, строкой где совпадает `line`/`block`-
правило, или концом ввода. Следствия:

- Пустая строка *внутри* открытой конструкции закрывает прогон и отбрасывает
  эту конструкцию: JSON-образная грамматика (пустые `lines`/`blocks`) не должна
  содержать пустых строк посреди значения.
- Приоритет между перекрывающимися инлайн-правилами по-прежнему следует порядку
  объявления, а не таблице приоритетов.

### Расхождение standalone vs полный парс

Standalone-итераторы (`find_*`) производят отличающиеся от полного парса
результаты в ряде случаев:

- Разделитель внутри ограждённого блока подавляется полным парсером
  (который отслеживает стек активных блоков), но совпадает в
  standalone-итераторе (у которого нет такого состояния).
- Экранированный разделитель подавляется проверкой escape в полном парсере,
  но может совпасть в standalone-итераторе если его логика экранирования
  отличается.
- Standalone-правила `symmetric`/`asymmetric` совпадают только с точным
  объявленным счётом и никогда не участвуют в ограниченной вложенности
  вообще, независимо от грамматического `max_nest` или собственных настроек
  `balanced`/`parse_inside` этого правила.

Это задокументировано и сделано намеренно. Standalone-итераторы обменивают
корректность на скорость в случае одного вида элементов.

### Ограничение входа в 4 ГБ

Спаны хранят байтовые смещения как `u32`. Входные данные больше `u32::MAX`
байт производили бы молча усечённые спаны. Fuzzing-harness защищает от этого
ранним выходом. На практике ни один текстовый документ не приближается к
этому пределу.

## 18. Точки расширения

### Добавление нового грамматического крейта

Создайте новый крейт, зависящий от `meon`. Вызовите `define_parser!` один раз
с новой грамматикой. Крейт становится полностью самодостаточным — имеет
собственные `NameParser`, `NameContent` и методы `find_*` без общего
изменяемого состояния.

### Добавление нового вида правил

Добавление нового вида правил требует изменений в обоих крейтах:

**meon-macros:**
1. Добавить вариант в `StandaloneRule` в `model.rs`.
2. Разобрать новый синтаксис в `collect.rs` (в соответствующей функции `collect_*`).
3. Испустить запись в standalone DSL в `codegen.rs` (`build_standalone_dsl`).
4. Испустить акцессоры `_clean` / `_raw` в `methods.rs`.

**meon:**
1. Добавить соответствующую ветвь в `define_standalone_fns!` в `standalone/mod.rs`.
2. Реализовать структуру итератора в новом файле в `standalone/`.
3. Ре-экспортировать структуру из `lib.rs`.
4. Добавить новый синтаксис в соответствующий рантайм-макрос (`parse_inline!`,
   `parse_line!` или `parse_block!`).

Если новый вид правил должен поддерживать вложенность, следуйте уже
существующему паттерну `balanced`/`parse_inside`, используемому
`symmetric`/`asymmetric` (§9), а не изобретайте новый механизм — переиспользуйте
грамматический предел `max_nest` и унифицированный стек `frames` с его единым
проходом закрытия, чтобы это корректно компоновалось с уже вкладывающимися
правилами.

### SIMD-бэкенды

Новые SIMD-бэкенды добавляются в `swar.rs` добавлением нового Cargo feature,
включением через `portable_simd` и добавлением вызова `search_simd!` внутри
`find_any_wide`. Диспатч N=1..3 на `memchr` не затрагивается.

### Замена аллокатора

`meon` использует глобальный аллокатор для всех выделений `Vec`. Замена его
на `mimalloc` или другой аллокатор — ответственность бинарного/грамматического
крейта, а не движка. Изменений в `meon` не требуется.
