# meon — Фузз-тестирование

[**EN**](https://github.com/vgnapuga/meon/blob/main/fuzz/README.md) | RU

Фузз-тестирование с покрытием для движка [`meon`](https://github.com/vgnapuga/meon/blob/main/meon/README.md)
с использованием [`cargo-fuzz`](https://github.com/rust-fuzz/cargo-fuzz)
(libFuzzer). Единственная цель прогоняет движок через **fuzz-only грамматику**,
собранную из набора правил [`meon-md`](https://github.com/vgnapuga/meon/blob/main/meon-md/README.md)
плюс одно правило `key_value` (см. [Цель](#цель)). Это **исключительно тест на
паники / безопасность памяти** — *корректность* грамматики живёт в наборах
юнит- и интеграционных тестов, не здесь.

## Цель

Единственная цель — `parse_text` (`fuzz/fuzz_targets/parse_text.rs`). Она
передаёт произвольные байты в fuzz-only парсер `MdKvFuzzParser` и держит движок
на **полу «не паникует / в границах»** по трём отдельным путям кода за один
прогон.

```rust
fuzz_target!(|data: &[u8]| {
    if data.len() > MAX_INPUT_LEN {
        return;
    }
    // Fuzz-only грамматика: meon-md дословно + одно правило `key_value`,
    // разделяющее унифицированный стек frames/fdepth (см. «Зачем fuzz-only
    // грамматика» ниже).
    let c = MdKvFuzzParser::parse(data);

    let check = |start: u32, end: u32| {
        assert!(start <= end);
        let _ = &c.source[start as usize..end as usize];  // паника при OOB
    };

    // (a) Корректность спанов полного парса, каждое поле — включая `pairs`:
    //     texts, bolds, italics, bold_italics, codes, autolinks,
    //     hard_breaks (start == end), links, pairs, paragraphs, blockquotes,
    //     fenced_codes, headings, thematic_breaks, bullet_items, ordered_items

    // (b) Сгенерированные аксессоры `_raw()` / `_clean()`, прогнанные до конца —
    //     их арифметика разделителей (`start - count` / `end + count`) и есть
    //     паника-опасная часть — для codes/italics/bolds/bold_italics/autolinks.

    // (c) Standalone-итераторы `find_*` по `data` — ОТДЕЛЬНЫЙ путь кодогена;
    //     без сравнения с полным парсом (они расходятся по дизайну),
    //     только тот же пол по границам.
});
```

### Зачем fuzz-only грамматика

Грамматика — **не** опубликованная грамматика `meon-md`. Это набор правил
`meon-md`, скопированный дословно, плюс ровно одно добавление: правило
`key_value` внутри того же блока `on_trigger`, что и `*` / `` ` `` / `<` / `[`,
с `:` как триггером `eq` (байт `end` — `,` — авто-добавляется движком).

Причина — покрытие, а не реализм. `meon-md` не объявляет правила `key_value`,
поэтому самая хитрая новая механика движка — `key_value`-фреймы, разделяющие
**унифицированный стек `frames` / `fdepth`** с `balanced = true` symmetric (`*`)
и asymmetric (`<`,`>`) фреймами, дренаж kv-перед-pop'ом в close-каскаде, дренаж
в конце прогона — получает от продакшен-грамматики **ноль** fuzz-покрытия. Эта
цель существует, чтобы гонять именно эти механизмы движка, комбинируя правила,
которые все попадают на один общий стек. Она намеренно *не* поставляется в
`meon-md`: срабатывание `:` на обычной прозе (`Note:`, `3:00`, URL после
`http`) на каждом реальном парсе — неприемлемая цена для продакшен-крейта ради
этого покрытия.

Грамматика привязана к **механизмам движка**, а не к `meon-md` — она валидна
ровно до тех пор, пока сохраняет комбинацию правил, упражняющую общий стек, и не
несёт обязательства отслеживать грамматику `meon-md` со временем.

### Что проверяет цель

Все три фазы — это один и тот же пол: *никакой паники, каждый спан в границах* —
и ни одна ничего не утверждает о том, *какими* спаны должны быть (это задача
юнит-/интеграционных наборов):

1. **Полный парс** — каждое поле `MdKvFuzzContent`, включая `pairs`; якоря
   жёстких переносов дополнительно проверяются на нулевую длину (`start == end`).
2. **Аксессоры `_raw()` / `_clean()`** — сгенерированная арифметика
   разделителей, отдельная забота от голых спан-полей, упражняется прогоном
   каждого итератора до конца. `_raw` расширяет хранимый спан наружу на длину
   прогона разделителя; underflow в `start - count` или `end + count` за
   пределами буфера паниковали бы здесь.
3. **Standalone-итераторы `find_*`** — испускаются *другой* ветвью макроса,
   нежели `parse`, сканируются по сырому `data`. Они задокументированно
   расходятся с полным парсом по содержимому, поэтому сравнения значений нет —
   только пол по границам.

## Проверяемые инварианты

Для каждого спана — из полного парса, из аксессора `_raw`/`_clean` или из
итератора `find_*`:

- `start <= end` — спаны являются корректными полуоткрытыми диапазонами.
- `source[start..end]` не паникует — спаны никогда не выходят за пределы входа.
- Якоря жёстких переносов строк нулевой длины: `start == end`.
- Пары `key_value` валидируют и `key`, и `value` спаны независимо.
- Входы длиннее `MAX_INPUT_LEN` (`u32::MAX`, 4 ГБ) пропускаются сразу,
  поскольку `u32`-смещения не могут их представить (см.
  [`ARCHITECTURE.md §14`](https://github.com/vgnapuga/meon/blob/main/ARCHITECTURE.md#14-span-representation)).

Замыкание `check` использует реальную индексацию среза (`&c.source[start..end]`),
а не числовое сравнение — поэтому любой выход за пределы вызывает немедленную
панику, которую libFuzzer перехватывает и сохраняет как воспроизводящий артефакт.

Это пол безопасности, а не оракул корректности. Цель намеренно **не** делает
ни одного утверждения о логике грамматики — что объект из двух членов
разрешается в два члена, что ключ пары предшествует её значению, что
`find_bolds` согласуется с полным парсом. Это покрыто юнит-тестами `meon` и
интеграционными тестами `meon-md` / `meon-json`; здесь единственный контракт —
«никакой вход, сколь угодно искажённый, не заставляет движок паниковать или
читать за границами».

## Требования

- **Nightly**-тулчейн (libFuzzer требует санитайзер-рантайм).
- `cargo-fuzz`.

Оба доступны в dev-шелле:

```sh
nix develop
```

За пределами Nix:

```sh
rustup toolchain install nightly
cargo install cargo-fuzz
```

## Запуск

```sh
# Список целей:
cargo fuzz list

# Запуск до прерывания (Ctrl-C):
cargo fuzz run parse_text

# С ограничением по времени:
cargo fuzz run parse_text -- -max_total_time=7200

# Без AddressSanitizer (в 2-4 раза быстрее; безопасно — крейт не содержит unsafe):
cargo fuzz run parse_text --sanitizer none -- -max_total_time=7200

# С сидами из бенчмарковых корпусов (рекомендуется — ускоряет рост покрытия):
cargo fuzz run parse_text fuzz/corpus/parse_text
```

## Триаж

```sh
# Повторить сохранённый краш:
cargo fuzz run parse_text fuzz/artifacts/parse_text/<crash-file>

# Минимизировать краш до наименьшего воспроизводящего входа:
cargo fuzz tmin parse_text fuzz/artifacts/parse_text/<crash-file>

# Прогнать корпус без фаззинга (регрессионная проверка):
cargo fuzz run parse_text fuzz/corpus/parse_text -- -runs=0
```

Папки `corpus/`, `artifacts/`, `coverage/` и `target/` игнорируются git
(`fuzz/.gitignore`).

|----------------|------------|--------------------|----------------|------|------|------------|--------|-------|

| v0.2.0         | 2026-06-21 | nightly-2026-05-22 | ~111M          | 1114 | 6853 | 2346/440Kb | ~32k   | 641Mb |

**Насыщение покрытия** на `cov: 1114 ft: 6853 corp: 1758/252Kb` означает что
libFuzzer исчерпал достижимые ветки на случайных входах без сидов. Добавление
сид-документов из реальных Markdown-файлов или из бенчмарковых корпусов поднимет
покрытие выше — фаззер получит точку входа в структурированные пути исполнения.
Добавленные в цель пути `key_value` / аксессоров / standalone расширяют
достижимую поверхность, так что точка насыщения на следующей кампании сместится.

## Примечания

- Крейт **не содержит `unsafe`-кода** (`#![forbid(unsafe_code)]`), и цель
  упражняет только движок и его сгенерированный парсер, поэтому AddressSanitizer
  для целей безопасности памяти избыточен и может быть отключён через
  `--sanitizer none` для ускорения в 2–4 раза.
- Standalone-итераторы `find_*` теперь **упражняются** этой целью (фаза (c)
  выше) — отдельный путь кодогена от `parse`, на том же полу по границам. Так же
  упражняются сгенерированные аксессоры `_raw()` / `_clean()` (фаза (b)).
- Цель не сверяет полный парс со standalone-итераторами: они расходятся по
  дизайну (разделитель внутри ограждения, экранированное закрытие), поэтому
  утверждается только собственный пол безопасности каждого пути.
- `avx512` не тестировался в ходе фаззинга — железо с AVX-512 было недоступно.
  Скалярный путь и путь `avx2` покрыты.
