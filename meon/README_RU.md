# meon

[**EN**](https://github.com/vgnapuga/meon/blob/main/meon/README.md) | RU

Декларативный плоский движок парсинга текстовых форматов.

Вы описываете грамматику один раз через `define_parser!` и получаете полноценный
парсер: метод `parse` для полного однопроходного сканирования и standalone-итераторы
`find_*` для ленивого извлечения одного вида элементов за раз.

```toml
[dependencies]
meon = "0.3"
```

## Быстрый старт

```rust
use meon::define_parser;

define_parser!(Plain {
    sep = b' ', eol = b'\n', tab = b'\t', escape = b'\\';

    inline {
        fallback => texts [10];
    }
    blocks {
        fallback => paragraphs [80];
    }
});

let content = PlainParser::parse(b"hello world\n");
assert_eq!(content.paragraphs.len(), 1);
assert_eq!(content.texts.len(), 1);
```

---

## Как это работает

`define_parser!(Name { ... })` разворачивается в:

- `NameContent<'a>` — выходная структура. Каждое правило грамматики производит одно `pub`-поле. Все поля заимствуют оригинальный срез источника.
- `NameParser` — unit-структура с методами:
  - `parse(source: &[u8]) -> NameContent<'_>` — полный однопроходной парс, O(n).
  - `find_*(source: &[u8]) -> impl Iterator` — по одному на правило, контекст-свободные, быстрее когда нужен только один вид элементов.
- `str(span) -> Option<&str>` / `bytes(span) -> &[u8]` и аксессорные методы
  `_clean` / `_raw` на `NameContent` для удобного преобразования спанов в срезы.

Спаны — `u32` байтовые смещения. Входные данные не должны превышать 4 ГБ (`span::MAX_INPUT_LEN`).

## Синтаксис грамматики

```
define_parser!(Name {
    sep = b' ', eol = b'\n', tab = b'\t', escape = b'\\';

    inline  { ... }
    lines   { ... }
    blocks  { ... }
});
```

Четыре контекстных байта обязательны и применяются к каждому правилу. Пятый,
`max_nest`, опционален и грамматический:

| Ключ       | Значение                                                       |
|------------|----------------------------------------------------------------|
| `sep`      | Разделитель слов (обычно пробел)                               |
| `eol`      | Конец строки (обычно `\n`)                                     |
| `tab`      | Символ табуляции                                               |
| `escape`   | Префикс экранирования, подавляющий следующий байт              |
| `max_nest` | Опционально. Ограниченный предел глубины само-вложенности,     |
|            | разделяемый между `balanced` symmetric/asymmetric правилами и  |
|            | значениями `key_value` (ниже) и блочной вложенностью           |
|            | `cont`/`fence`. По умолчанию `1`, что точно воспроизводит      |
|            | исходное, не-вкладывающееся поведение: никакого само-вложения, |
|            | максимум один активный блок.                                   |
|            | `sep = ..., eol = ..., tab = ..., escape = ..., max_nest = 4;` |

## Семейства правил

Правила делятся на три семейства в зависимости от того, где в источнике элемент может начаться и закончиться.

### Инлайн-правила — внутри строки

Инлайн-элементы находятся сканированием триггерных байт внутри содержимого строки. Два подвида:

**`inline_simple`** — единственный `Span` без дополнительных метаданных.
Объявляется через `fallback` или как ветви совпадения внутри `on_trigger`.
Тип поля: `Vec<Span>`.

**`inline`** — пользовательская структура с несколькими `Span`-полями.
Объявляется через `chained` или `key_value` внутри `on_trigger`.
Тип поля: `Vec<T>`.

---

#### `on_trigger(b1, b2, ...) { ... }`

Объявляет набор триггерных байт. Когда любой из них находится в строке —
блок активируется и правила пробуются по порядку объявления.

---

##### `symmetric byte { ... }` — одинаковый открывающий и закрывающий разделитель

```
on_trigger(b'*') {
    symmetric b'*' {
        parse_inside = true;   // режим pending: первый прогон открывает, следующий закрывает
        balanced     = true;   // само-вложение вхождений с разным счётом до max_nest
        1 => italics  [40],    // count => field [делитель_ёмкости]
        2 => bolds    [40],
        3 => bold_italics [80],
    }
}
```

- `parse_inside = true` — сканер запоминает открывающий прогон и закрывается на
  следующем совпадении по счёту (режим `pending`). Содержимое между открытием и
  закрытием само сканируется на инлайн-элементы — именно это позволяет конструкции
  вида `**жирный *курсив* текст**` опознать вложенный курсив вообще.
- `parse_inside = false` — сканер сразу ищет вперёд закрывающий прогон (жадный
  режим, используется для кодовых спанов). Содержимое внутри непрозрачно для
  любого другого правила.
- Значение `balanced` зависит от `parse_inside`:
  - При `parse_inside = true`: `balanced = false` означает что вхождение с другим
    счётом того же байта, пока одно уже pending, перезатирает pending-слот —
    никакого вложения, точно как исходное поведение с единственным слотом.
    `balanced = true` вместо этого открывает *ограниченный стек* pending-фреймов
    (размером по грамматическому `max_nest`), так что вхождение с другим счётом
    открывает собственный фрейм вместо перезатирания внешнего — оба уровня
    разрешаются и эмитятся в свои поля отдельно. *Идентичная* пара `(byte, count)`
    всё ещё не может само-вложиться, поскольку открытие и закрытие выглядят
    одинаково для симметричного разделителя (`**a **b** c**` резолвится как два
    соседних прогона, не как вложение).
  - При `parse_inside = false` (жадный режим): `balanced = true` вместо этого
    означает что *удвоенный* прогон того же разделителя, найденный при поиске
    вперёд, считается экранированным/буквальным содержимым а не закрытием, и
    поиск продолжается дальше — не связано со стеком выше.
- Каждая ветвь `N => field [div]` захватывает прогоны ровно из N байт → `Vec<Span>`.
- `_ => field [div]` захватывает любой счёт не совпавший с явными ветвями.

---

##### `asymmetric open, close { ... }` — разные открывающий и закрывающий байты

```
on_trigger(b'{', b'}') {
    asymmetric b'{', b'}' {
        balanced     = true;
        parse_inside = false;
        1 => objects [100],
    }
}
```

- `balanced` и `parse_inside` — независимые настройки, любой из них одной
  достаточно чтобы попасть на ограниченный стек, размером по грамматическому
  `max_nest`:
  - `balanced` задаёт собственный эффективный предел глубины для типа:
    `max_nest` если `true` (так что `{ { } }` само-вкладывается), или жёсткая
    `1` если `false` — второе открытие того же типа пока одно уже pending
    тогда просто буквальное, точно как исходное поведение для этого типа.
  - `parse_inside` контролирует *непрозрачность*: `false` оставляет содержимое
    между открытием и закрытием невидимым для любого другого правила
    (используется для автоссылок, чтобы собственные `(`/`)` url не были приняты
    за что-то другое); `true` делает его прозрачным, так что другие правила —
    включая разные типы скобок на этом же стеке — могут срабатывать на байтах
    между ними.
- Закрывающий байт сопоставляется с *собственным записанным* открывающим байтом
  фрейма, не с тем, какое правило проверяется первым — так что два разных
  asymmetric-правила, объявленных в одном блоке `on_trigger`, могут безопасно
  делить закрывающий байт (например, `(`/`)` и `[`/`)`), пока их открывающие
  байты различаются.
- Как только `balanced` или `parse_inside` равны `true`, закрывающий байт
  **обязан** быть в том же наборе `on_trigger(...)` что и открывающий —
  `on_trigger(b'{', b'}')`, не просто `on_trigger(b'{')` — поскольку закрытие
  теперь находится тем же сканированием что находит открытие, не отдельным
  форвард-поиском.
- Ветви счёта работают так же как в `symmetric`, кроме того что каждый байт
  многобайтового прогона — своё собственное событие: `{{` — это два открытия,
  не одно событие "счёт = 2".

---

##### `chained: Type { ... }` — двухчастный разделитель (например, ссылки)

```
on_trigger(b'[') {
    chained: Link {
        | b'[', b']' | { parse_inside = false; balanced = false; } => text,
        | b'(', b')' | { parse_inside = false; balanced = false; } => url,
        prefix | b'!' | => is_image,
    } => links [100]
}
```

Совпадает с паттерном `[prefix]open1...close1 open2...close2`.

- Две пары `| open, close |` определяют два компонента.
- `prefix | byte |` объявляет необязательный одиночный байт непосредственно перед `open1`, который устанавливает булево поле.
- `parse_inside = true` на любом компоненте делает этот компонент прозрачным —
  другие правила могут срабатывать на байтах, через которые сканирование
  прошло для него — вместо исходного, полностью непрозрачного самодостаточного
  форвард-поиска. Это ограничено одним активным `chained`-правилом на
  грамматику; грамматика с двумя такими правилами и перекрывающимися
  незавершёнными совпадениями не поддерживается.
- Тип поля: `Vec<T>`.

---

##### `key_value: Type { ... }` — пары `ключ = значение`

```
on_trigger(b'=') {
    key_value: KeyValue {
        eq        = b'=';
        allow_sep = true;   // обрезать пробелы вокруг eq
        end       = b'\n';  // значение заканчивается здесь (или на eol)
        key   => key,
        value => value,
    } => key_values [20]
}
```

- Байт `eq` **обязан** быть в наборе `on_trigger(...)`; байт `end`
  добавляется в этот набор автоматически — самому указывать его не нужно.
- Значение участвует в общем ограниченном стеке вложенности: оно переживает
  вложенные `symmetric`/`asymmetric` контейнеры, открытые внутри него (до
  грамматического `max_nest`), и переживает внутренние байты `eol` внутри
  прогона. Пара эмитится только когда её значение реально закрывается — на
  `end`, на закрытии контейнера, в котором лежит значение, или в конце
  прогона — поэтому `Vec<T>` сохраняется в порядке закрытия значений.
- Выходной тип `T` должен быть определён с полями, названными по
  идентификаторам `=> key` / `=> value`. Тип поля: `Vec<T>`.
- Поддерживается только одно `key_value`-правило на грамматику (см. Известные
  ограничения).

---

##### `fallback => field [div]` — простой текст

Захватывает прогоны байт, не сработавшие ни на одно другое инлайн-правило.

```
fallback => texts [10];
```

Соседние спаны могут объединяться (разрыв ≤ 1 байт) установкой `merge_simple = true`
в начале секции `inline`.

---

##### `hard_break(esc, sp, min) => field [div]` — жёсткий перенос строки

```
hard_break(b'\\', b' ', 2) => hard_breaks [500];
```

Обнаруживает маркер жёсткого переноса в конце строки: либо байт `esc`, либо
не менее `min` подряд идущих байт `sp`. Эмитирует `Span` нулевой длины.

---

### Строчные правила — вся строка потребляется

Совпавшая строка потребляется целиком; инлайн-сканирование для неё пропускается.
Оба вида строчных правил производят `Vec<(Type, Span)>` где `Span` покрывает
содержимое после маркера.

---

##### `line(byte, max = N) |var|: Type { ... } => field [div]`

Совпадает со строками начинающимися с 1–N последовательных вхождений `byte`
за которыми следует `sep` или конец строки. `var` получает счёт.

```
line(b'#', max = 6) |n|:
    Heading { level: NonZeroU8::new(n).unwrap_or(NonZeroU8::MIN) }
    => headings [200];
```

---

##### `line_simple(b1 | b2 | ..., min = N) |var|: Type { ... } => field [div]`

Совпадает со строками целиком состоящими из одного повторяющегося байта-разделителя
(перемежаемого с `sep`), встречающегося не менее `min` раз. `var` получает байт разделителя.

```
line_simple(b'-' | b'*' | b'_', min = 3) |b|:
    ThematicBreak { kind: b }
    => thematic_breaks [200];
```

---

### Блочные правила — охватывают несколько строк

Блочные элементы начинаются на одной строке и заканчиваются на более поздней.

---

#### `block_simple { ... }` — многострочные спаны без метаданных на строку

Тип поля: `Vec<Span>`.

##### `fence(byte, min = N) => field [div]`

Открывается когда строка начинается с ≥ N последовательных байт `byte` и не
содержит дальше `byte` в информационной строке. Закрывается на следующей строке
начинающейся с не менее стольких же байт `byte`, за которыми следуют только `sep`
или `tab`. Весь диапазон (от открывающего ограждения до закрывающего включительно)
— один `Span`. **Инлайн-сканирование подавляется пока ограждение активно.**

```
fence(b'`', min = 3) => fenced_codes [400];
```

##### `cont(byte) => field [div]`

Группирует последовательные строки начинающиеся с `byte` в единственный `Span`.
Закрывается когда строка не начинается с `byte`. При `max_nest > 1` `cont` может
само-вложиться — `> > текст` открывает два различных, корректно ограниченных
спана, поскольку маркер проверяется позиционно на каждом снятии, и ещё один
`cont` может открыться сразу после на той же строке.

```
cont(b'>') => blockquotes [200];
```

---

#### `block { ... }` — поэлементные записи с метаданными

Тип поля: `Vec<(Type, Span)>` — по одной записи на каждую совпавшую строку.

##### `(pattern) |var|: Type { ... } => field [div]`

Совпадает со строками где после необязательных ведущих пробелов одиночный байт
удовлетворяющий `pattern` сопровождается `sep` или `tab`. `var` получает байт маркера.

```
(b'-' | b'*' | b'+') |b|:
    BulletItem { kind: b }
    => bullet_items [80];
```

##### `num(digit_pat, end = end_pat) |n, k|: Type { ... } => field [div]`

Совпадает со строками где прогон цифр (до 9 цифр) сопровождается байтом
удовлетворяющим `end_pat` а затем `sep` или `tab`. `n` получает разобранное число,
`k` — байт разделителя.

```
num(b'0'..=b'9', end = b'.' | b')') |n, k|:
    OrderedItem { kind: k, num: n }
    => ordered_items [80];
```

##### `fallback => field [div]`

Строки не совпавшие ни с одним другим блочным правилом группируются в спаны параграфов.

```
fallback => paragraphs [80];
```

## Инлайн-прогоны охватывают несколько строк

Инлайн-сканирование работает не построчно. Прогон строк, не совпавших ни с
одним правилом `line`/`block`/`fence`/`cont`, передаётся инлайн-сканеру как
**один** многострочный спан. Внутри этого прогона `eol` — обычное
содержимое: открытый фрейм `symmetric`/`asymmetric` или незавершённое
значение `key_value` переживает перенос строки и закрывается только в
настоящем конце прогона.

Прогон ограничивается — и инлайн-стек сливается — на пустой строке, на
строке где совпадает правило `line`/`block`, или в конце ввода. Поэтому
грамматика с пустыми секциями `lines {}` / `blocks {}` (например,
JSON-образная) получает один прогон на весь ввод (помимо пустых строк), и её
вложенность переживает каждый внутренний `\n` без дополнительной стоимости.
Грамматика с реальными правилами `lines`/`blocks` сохраняет прежнее
ограничение по параграфам: многострочное инлайн-перекрытие случается только
внутри того, что и так было бы одним параграфом.

## Делители ёмкости

Каждое поле несёт `[div]`. Начальная ёмкость `Vec` равна `source.len() / div`.
Делитель `10` означает примерно один элемент на 10 байт. Настраивайте исходя
из ожидаемой плотности элементов в реальных входных данных.

## Standalone-итераторы

Каждое правило которое это поддерживает генерирует метод `find_*`:

```rust
// вместо полного парса:
let content = MyParser::parse(source);

// сканируем только один вид элементов:
for span in MyParser::find_italics(source) {
    println!("{}", std::str::from_utf8(&source[span.start as usize..span.end as usize]).unwrap());
}
```

Standalone-итераторы сканируют сырой источник без какого-либо межэлементного
контекста. Они могут возвращать спаны которые полный парсер подавил бы (например,
разделители внутри ограждения). Счётчики могут отличаться от полного парса —
это намеренно. Standalone-правила `symmetric`/`asymmetric` совпадают только с
точным объявленным счётом и никогда не участвуют в ограниченном стеке
вложенности, независимо от грамматического `max_nest` или собственных настроек
`balanced`/`parse_inside` этого правила.

## Feature flags

| Feature  | Эффект                                                    |
|----------|-----------------------------------------------------------|
| `avx2`   | 32-байтовый SIMD-поиск (требует nightly + AVX2 процессор) |
| `avx512` | 64-байтовый SIMD-поиск (включает `avx2`)                  |

Без обоих флагов крейт компилируется на стабильном Rust с SWAR-fallback
(SIMD в регистре) для многобайтового поиска.

## Известные ограничения

- Само-вложенность (цитаты/ограждения на блочном уровне, `balanced`
  symmetric/asymmetric правила на инлайн-уровне) ограничена грамматической
  настройкой `max_nest` — по умолчанию `1`, то есть никакого само-вложения
  вообще. Конструкции вложенные глубже `max_nest` не отслеживаются специально;
  см. таблицу контекстных байт выше.
- `chained`-правило с прозрачным компонентом (`parse_inside = true`)
  отслеживает только одно незавершённое совпадение на грамматику; два таких
  правила с перекрывающимися совпадениями не поддерживаются.
- Инлайн-сканирование идёт по многострочному прогону (см. «Инлайн-прогоны
  охватывают несколько строк»), а не по одной строке: общий стек сохраняется
  через внутренние байты `eol`. Прогон по-прежнему ограничен пустой строкой,
  поэтому пустая строка *внутри* открытой конструкции закрывает прогон и
  отбрасывает эту конструкцию — JSON-образная грамматика не должна содержать
  пустых строк посреди значения. Приоритет между перекрывающимися
  инлайн-правилами определяется порядком объявления, а не таблицей
  приоритетов.
- Поддерживается только одно `key_value`-правило на грамматику; его якорь
  сегмента ключа разделяемый, поэтому второе `key_value`-правило не
  отслеживается независимо.
